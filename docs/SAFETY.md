# 拔插安全与数据完整性

## 引擎对 SSD 做什么、不做什么

| 文件 | 打开方式 | 说明 |
|---|---|---|
| `out-*.safetensors`（370GB 模型分片） | **只读**（`O_RDONLY`，上游 `c/st.h`） | 任何时候都不会被写入。硬拔、断电、进程被杀——模型都不会坏 |
| `.coli_kv` | 追加写（~182KB/token） | 会话 KV 缓存，append-only、crash-safe 设计。`KVSAVE=0` 完全关闭 |
| `.coli_usage` | 小量写 | 学习缓存：记录专家路由习惯，下次启动自动 pin 热专家。**它跟着 SSD 走，所以这块盘插到哪台机器都记得你的用法** |

结论：**这块盘的负载本质是"99.99% 读 + KB 级追加写"**，对 SSD 寿命的影响可以忽略（上游 README 原话：reads don't meaningfully wear an SSD）。

## 三种使用姿势

1. **默认**：正常退出聊天（`:exit` / Ctrl-D），再安全弹出。状态文件正常累积，模型越用越快。
2. **纯只读模式**：`./start.sh --readonly`（= `KVSAVE=0`）。全程零写入，硬拔也无可损失——代价是不保存会话、不积累学习缓存。
3. **运行中硬拔**（不推荐但死不了盘）：进程异常退出；模型分片只读无损；`.coli_kv` 是 crash-safe 的，最多丢最后一轮对话状态。文件系统层面 NTFS/ext4 都有日志，重新挂载做一致性检查即可。FAT/exFAT 无日志，意外拔掉风险高——又一个别用 exFAT 的理由。

## 真正的风险不是拔盘，是这两个

1. **宿主机内存耗尽触发 swap**：swap 会把内存页写回盘上，和引擎的专家读取抢带宽，还会真实磨损 SSD。`start.sh` 的内存门槛（默认 16GB 起步、建议 25GB）就是为了躲开这个坑，别随便用 `COLI_MIN_RAM_GB=0` 硬闯。
2. **持续满载的散热**：几小时满速读取会让廉价硬盘盒/QLC 盘过热降速。选带散热的盒子（见 HARDWARE.md）。

## Windows 特别项：杀毒软件

Defender 实时扫描 370GB 模型分片会让首次加载和专家流读慢得离谱。**首次使用前执行一次（管理员 PowerShell）**：

```powershell
Add-MpPreference -ExclusionPath "X:\model\glm52_i4"
```

同理，第三方杀毒/"安全卫士"类软件也要加排除目录。

## 跨机器状态一致性

`.coli_kv` 里存有模型版本校验——上游会拒绝加载"来自不同模型或版本"的 KV 文件（`glm.c`），所以换机器、换引擎版本后旧会话不会错配，最坏情况只是重新 prefill。`.coli_usage` 是统计直方图，跨机器累积无副作用。
